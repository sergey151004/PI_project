# Отчёт по тестированию проекта PI_project

## 1. Методологическая основа

В отчёте используются критерии из конспекта презентаций:

- **Ценность самотестирующегося кода:** тесты автоматические, их можно запускать часто; рефакторинг опирается на тесты.
- **Виды тестов:** юнит-тесты (проверка фрагмента кода в изоляции, внутреннее качество) и функциональные (система как «чёрный ящик», внешнее качество).
- **Свойства хорошего юнит-теста:** повторяемость, независимость, очевидность назначения, быстрота, своевременность.
- **Структура AAA (Arrange–Act–Assert):** подготовка данных → вызов метода → проверка результата.
- **Именование:** желательно отражать состояние/поведение (например, `MethodName_StateUnderTest_ExpectedBehavior` или `Should_ExpectedBehavior_When_StateUnderTest`).
- **Антипаттерны:** тестирование приватных методов, утечка доменных знаний в тесты, загрязнение продуктивного кода тестовым.

Ниже по каждому сервису разобраны существующие тесты и даны выводы.

## 2. courses_service

**Файл:** `courses_service/courses/tests.py`

### 2.1 Что проверяется

| Класс | Тест | Объект проверки |
|-------|------|------------------|
| `CourseModelTest` | `test_create_course` | Создание модели `Course`, сохранение полей `name` и `description`. |
| `CourseSerializerTest` | `test_course_serializer` | Сериализация `Course` через `CourseSerializer` — совпадение полей в выводе. |

### 2.2 Разбор по методологии

**Вид тестов:** юнит-тесты — проверяются отдельно модель и сериализатор (небольшие фрагменты кода).

**Структура AAA:**

- `test_create_course`: **Arrange** — нет явной подготовки (создание в Act); **Act** — `Course.objects.create(...)`; **Assert** — `assertEqual` по полям. Логика соответствует AAA, но подготовка и действие объединены в одном вызове.
- `test_course_serializer`: **Arrange** — создание курса; **Act** — создание сериализатора и обращение к `.data`; **Assert** — проверка полей. Структура соблюдена.

**Именование:** имена вида `test_create_course` и `test_course_serializer` не отражают состояние и ожидаемое поведение. По конспекту предпочтительнее, например: `create_course_WithValidData_SavesNameAndDescription` или `course_serializer_WithValidCourse_ReturnsMatchingFields`.

**Независимость и повторяемость:** Django `TestCase` откатывает БД после каждого теста, поэтому тесты не зависят друг от друга и от порядка запуска — свойство **независимости** и **повторяемости** соблюдено.

**Замечания:**

- Дублирование импорта `from django.test import TestCase` в начале файла — лишний код.
- Нет тестов API (например, `GET/POST /api/courses/`): эндпоинты не покрыты; при этом в корневом `urls.py` курсов API может быть не подключён — тесты на view/API могли бы это выявить.
- Один тест — один логический блок AAA в целом соблюдается; в Act только один значимый вызов.

**Итог по courses_service:** тесты модели и сериализатора полезны, соответствуют идее юнит-проверок и структуре AAA. Именование и покрытие API можно улучшить.

## 3. grades_service

**Файл:** `grades_service/grades/tests.py`

### 3.1 Что проверяется

| Класс | Тест | Объект проверки |
|-------|------|------------------|
| `GradeModelTest` | `test_create_grade` | Создание модели `Grade`, значения `student_id`, `course_id`, `grade`. |
| `GradeSerializerTest` | `test_grade_serializer` | Сериализация `Grade` — поля в `data`. |
| `GradeAPITest` | `test_get_grades` | GET `/api/grades/` — код 200 и количество записей в ответе. |
| `GradeAPITest` | `test_create_grade` | POST `/api/grades/` — код 201 и увеличение числа записей в БД. |

### 3.2 Разбор по методологии

**Виды тестов:** модель и сериализатор — юнит-тесты; `GradeAPITest` — функциональные (проверка HTTP-эндпоинтов, система как чёрный ящик).

**Структура AAA:**

- `test_create_grade` / `test_grade_serializer`: Arrange (создание объекта) → Act (операция) → Assert — соответствуют.
- `test_get_grades`: **Arrange** — `setUp()` создаёт одну оценку; **Act** — `self.client.get("/api/grades/")`; **Assert** — статус и длина `response.data`. Один логический блок, AAA соблюдён.
- `test_create_grade`: **Arrange** — словарь `data`; **Act** — `post`; **Assert** — статус 201 и `Grade.objects.count() == 2`. Проверяется и выход (код ответа), и **состояние** системы (количество записей) — в духе конспекта «проверка состояния».

**Независимость:** в `GradeAPITest` общие данные в `setUp()` создают одну запись; второй тест добавляет ещё одну и проверяет count. Благодаря транзакциям Django тесты изолированы, порядок запуска не влияет — повторяемость и независимость соблюдены.

**Именование:** имена общие (`test_get_grades`, `test_create_grade`). По конспекту лучше отражать сценарий, например: `get_grades_WhenGradesExist_ReturnsListAnd200` или `create_grade_WithValidData_Returns201AndIncrementsCount`.

**Замечания:**

- Жёстко заданный путь `"/api/grades/"`: при смене URL в проекте тесты сломаются. Предпочтительнее `reverse()` по имени маршрута.
- Нет проверки фильтрации по `student_id` (если такой функционал планируется или уже есть в коде).
- Формула эффективности из конспекта (защита от багов × устойчивость к рефакторингу × обратная связь × простота поддержки) в целом выполняется: тесты дают быструю обратную связь и защищают от регрессий при изменении API.

**Итог по grades_service:** хорошее сочетание юнит-тестов (модель, сериализатор) и функциональных (API). Структура AAA и изоляция соблюдены; улучшить именование и убрать захардкоженный путь.

## 4. users_service

**Файл:** `users_service/users/tests.py`

### 4.1 Что проверяется

| Класс | Тест | Объект проверки |
|-------|------|------------------|
| `UserModelTest` | `test_create_user` | Создание пользователя, поля `username`, `role`, проверка пароля. |
| `UserSerializerTest` | `test_user_serializer` | Сериализация пользователя — поля в выводе. |
| `UserListAPITest` | `test_user_list_requires_auth` | GET списка пользователей без авторизации — 401. |
| `UserListAPITest` | `test_user_list_authenticated` | GET списка с `force_authenticate` — 200. |
| `LoginViewTest` | `test_login_success` | POST на логин с верными данными — редирект 302. |
| `LoginViewTest` | `test_login_fail` | POST с неверным паролем — 200 (остаёмся на форме). |
| `GradesViewTest` | `test_grades_view` | GET `/grades/` после логина при замоканном `requests.get` — 200. |

### 4.2 Разбор по методологии

**Виды тестов:** модель и сериализатор — юнит; API и логин/страница оценок — функциональные (проверка сценариев для «заказчика»).

**Структура AAA:**

- В `UserModelTest` и `UserSerializerTest`: Arrange (создание User) → Act (сериализатор/данные) → Assert — соблюдена.
- В `UserListAPITest`: подготовка пользователя в `setUp`, в тестах — один вызов (get с клиентом) и проверка кода — AAA и один вызов в Act соблюдены.
- В `LoginViewTest`: подготовка в `setUp`, в тестах один `post` и проверка статуса — то же самое.
- В `GradesViewTest`: **Arrange** — логин и мок `requests.get`; **Act** — `get(reverse("grades"))`; **Assert** — 200. Используется **тестовый двойник (мок)** для внешней зависимости (HTTP к courses/grades), что соответствует конспекту: изоляция от внешнего API, проверка поведения view.

**Именование:** часть имён уже отражает сценарий: `test_user_list_requires_auth`, `test_login_success`, `test_login_fail` — понятно, что проверяется. Можно усилить стиль конспекта, например: `user_list_WhenUnauthenticated_Returns401`, `login_WhenValidCredentials_Redirects302`.

**Независимость и повторяемость:** у каждого класса свой `setUp`; Django создаёт изолированную БД для тестов и откатывает её. Тесты не зависят от порядка запуска и друг от друга.

**Сильные стороны:**

- Проверка и успешного, и неуспешного логина — граничные случаи.
- Проверка доступа к API без авторизации (401) и с авторизацией (200) — важный сценарий безопасности.
- Использование `reverse()` для URL и `force_authenticate` — хорошая практика, не привязка к строкам.
- Мок `requests.get` в `GradesViewTest` — корректное использование тестового двойника для внешней зависимости.

**Замечания:**

- В `test_grades_view` не проверяется контекст шаблона (передаются ли `courses` и `grades` из мока в шаблон). Можно добавить `assertContains` по ожидаемому контенту или проверку контекста.
- Закомментированный шаблон «Create your test here.» в начале файла — лишний комментарий.
- Дублирование импорта `TestCase` (как в courses).

**Итог по users_service:** тесты хорошо закрывают модель, сериализатор, авторизацию API и логин, а также страницу оценок с моком внешних запросов. Соответствуют идее «сначала код, потом тесты» и дают защиту при рефакторинге. Улучшения — именование в едином стиле и расширение проверок страницы оценок.

## 5. news_service

**Файлы:** `news_service/news/tests/test_factory.py`, `test_facade.py`, `test_views.py`

### 5.1 Что проверяется

| Файл / класс | Тест | Объект проверки |
|--------------|------|------------------|
| `test_factory.py` / `NewsFactoryTest` | `test_create_news_should_create_news_object` | Фабрика создаёт запись `News` с заданными полями; счётчик записей = 1. |
| `test_facade.py` / `NewsFacadeTest` | `test_create_news_should_create_news` | Фасад создаёт новость, проверяется заголовок. |
| `test_facade.py` / `NewsFacadeTest` | `test_get_news_should_return_news_list` | Две новости через фасад → `get_news()` возвращает список длины 2. |
| `test_views.py` / `NewsViewTest` | `test_news_page_should_return_200` | GET `/news/` — код 200. |
| `test_views.py` / `NewsViewTest` | `test_news_page_should_display_news` | Создание новости в БД, GET `/news/` — в ответе есть текст заголовка. |

### 5.2 Разбор по методологии

**Виды тестов:** фабрика и фасад — юнит-тесты (отдельные «части» логики); вьюха и HTTP — функциональные.

**Структура AAA:**

- **test_factory:** Arrange — нет явных данных кроме аргументов; Act — `NewsFactory.create_news(...)`; Assert — поля объекта и `News.objects.count() == 1`. Один вызов в Act, структура соблюдена.
- **test_facade (create):** Act — `NewsFacade.create_news(...)`; Assert — заголовок. Кратко и понятно.
- **test_facade (get_news):** Arrange — два вызова `create_news`; Act — `NewsFacade.get_news()`; Assert — `len(news_list) == 2`. Соответствует AAA.
- **test_views:** в первом тесте Act — `self.client.get("/news/")`, Assert — статус; во втором — Arrange (создание News), Act (get), Assert (`assertContains`). Один логический блок в каждом тесте.

**Именование:** использован стиль **Should_ExpectedBehavior** (частично): `test_create_news_should_create_news_object`, `test_news_page_should_return_200`, `test_news_page_should_display_news`. По конспекту это допустимый вариант именования; назначение теста понятно.

**Независимость:** каждый тест создаёт свои данные; Django `TestCase` изолирует БД. Тесты не зависят от порядка выполнения.

**Проверка состояния:** в тесте фабрики проверяется не только возвращаемый объект, но и состояние БД (`News.objects.count()`), что соответствует «проверке состояния» из конспекта.

**Замечания:**

- В `test_news_page_should_display_news` создание новости делается напрямую через `News.objects.create`, а не через `NewsFacade`. Для юнит-теста вьюхи это допустимо (проверяем отображение при наличии данных), но с точки зрения единообразия можно создавать через фасад, если важно проверять цепочку «фасад → БД → отображение».
- Путь `"/news/"` захардкожен; при смене URL тесты сломаются. Желательно `reverse()` по имени маршрута, если имена заданы в `urls.py`.
- Файл `tests/__init__.py` пустой — этого достаточно для обнаружения тестов Django; замечаний нет.

**Итог по news_service:** тесты покрывают ключевые единицы поведения (Factory, Facade) и сценарий отображения страницы. Структура AAA и независимость соблюдены; именование в стиле «should» понятное. Небольшие улучшения — использование `reverse()` и при необходимости создание данных через фасад в тесте вьюхи.

## 7. Соответствие принципам конспекта

- **Самотестирующийся код:** тесты автоматические, запускаются одной командой (`manage.py test`), результат однозначен (успех/падение). Рефакторинг можно проводить с опорой на тесты.
- **Подход «сначала код, потом тесты»:** в проекте тесты написаны для уже существующего кода, что соответствует указанному в конспекте подходу курса.
- **Повторяемость и независимость:** везде используется Django `TestCase` с откатом БД; общих изменяемых состояний между тестами нет — результат не зависит от порядка запуска.
- **Очевидность:** у большинства тестов назначение можно понять по имени класса и метода; в news_service именование в стиле «should» усиливает понятность.
- **Быстрота:** тесты не обращаются к внешним сервисам (в users мок заменяет реальные HTTP-запросы), БД — тестовая — выполнение быстрое.
- **Один тест — один логический блок AAA:** в разобранных тестах в секции Act, как правило, один значимый вызов; условий в теле теста нет — соответствует правилам «хорошего тона» из конспекта.
- **Антипаттерны:** приватные методы не тестируются; тестовый код вынесен в отдельные модули; продуктивный код не загрязнён тестами.
